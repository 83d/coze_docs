# 渠道入驻概述

扣子编程提供标准的生态对接方式，支持网站、App、应用市场、手机等硬件厂商作为自定义渠道入驻。在扣子编程中搭建并发布智能体到自定义渠道之后，智能体用户可以在新的发布渠道中和智能体对话。

## 渠道入驻介绍

在扣子编程中搭建智能体之后，如何将智能体推广到更多的用户群体中？扣子编程提供了飞书、豆包等各种不同的发布渠道，便于你的智能体触达用户，为各种 App 的用户提供服务。除此之外，扣子编程还支持自定义渠道入驻，你可以将智能体发布到搭建的个人网站或应用中，和你的网站或 App 绑定，为用户提供更加丰富的智能服务，提升用户体验。

扣子编程提供标准的生态对接方式和多种对接方案，支持网站、App、应用市场、手机等硬件厂商作为自定义渠道入驻。智能体发布到自定义渠道后，可以使用此渠道专属的 OAuth Access Token 调用扣子编程 OpenAPI，例如查看智能体信息、和智能体对话等。此外扣子编程还支持打通渠道侧的用户账号体系，在渠道中以渠道用户身份发布智能体、提供智能体对话服务。

## 入驻方案

扣子编程提供以下两种入驻方案，区别在于是否需要获取智能体开发者和渠道用户的对应关系，也就是说扣子编程和渠道侧的账号体系是否互通。

| **维度** | **账号隔离** | **账号互通** |
| --- | --- | --- |
| **方案详情** | 扣子编程和渠道侧的账号体系隔离，渠道侧无法查看智能体开发者和渠道用户的对应关系。 | 扣子编程和渠道侧的账号体系互通，扣子编程提供智能体开发者和渠道用户的对应关系，渠道用户可以自行管理自己的智能体。  此外，开发者必须授权扣子编程服务获取自己在渠道中的用户 ID，否则无法发布到渠道中。 |
| **应用场景** | 适用于业务架构较为简单的应用服务，例如个人自建网站、应用程序等。此类应用通常由平台侧提供智能体服务、统一管理智能体，无需打通渠道中的用户账号体系。 | 适用于智能体应用市场、手机厂商等渠道侧开发者多人共建、协同开发的场景，渠道侧的用户需要以自己的身份而不是渠道平台侧的身份维护自己的智能体。 |
| **实现难度** | 简单  只需配置 JWT 授权流程，并添加发布渠道。成功发布智能体后就可以通过渠道服务端调用扣子编程 OpenAPI 与智能体对话。 | 复杂  相较于账号隔离方案，账号互通方案要求渠道服务端具备 OAuth Server 能力、必须接收并处理消息回调、接入审核流程。 |

> [!NOTE]
>
> **说明**
>
> 所有扣子套餐均可创建自定义渠道，创建的自定义渠道默认为私有渠道，仅渠道创建者指定的内部工作空间可见。另外，扣子**企业版旗舰版**支持将渠道设置为公开，允许所有开发者发布智能体到渠道中，[单击](https://bytedance.sg.larkoffice.com/share/base/form/shrlgcE3ieqZw9kjpWoF5bcllih)[此处提交申请。](https://bytedance.sg.larkoffice.com/share/base/form/shrlgcE3ieqZw9kjpWoF5bcllih)

## 常见问题

1. 是否支持渠道审核流程？

支持。渠道如果设置了回调地址，表示接收渠道相关事件回调。开发者发布智能体到自定义渠道时，扣子编程会向渠道指定的回调地址发送智能体发布事件的回调消息，渠道侧收到智能体发布事件回调后，可以启动审核流程，并在审核结束后通过回调消息向扣子编程发送审核结果。

2. 渠道入驻和应用直接接入扣子编程 API 有什么区别？

- 在渠道入驻场景下，同一空间中的多个开发者均可以发布智能体到渠道中，共同构建渠道的智能体生态，一个统一的 OAuth Access Token 可以管理渠道中所有开发者发布的智能体。支持打通渠道账号体系，渠道中每个用户都可以维护自己的智能体。在智能体as API 场景下，一个 Access Token 只能使用开发者本人拥有的空间下的智能体。

- 渠道入驻场景和应用直接接入扣子编程可调用的扣子编程 API 范围一致。但在渠道入驻场景下，渠道侧在申请 OAuth Access Token 时，可以通过 scope 参数指定令牌的操作权限，例如此令牌只能和智能体 A 对话，不能执行其他操作。
